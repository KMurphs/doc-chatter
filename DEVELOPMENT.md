## Testing

There are three ways to test, each with different tradeoffs. All three call real Bedrock — you always need valid AWS credentials.

### 1. cargo lambda watch (fastest iteration)

Compiles and runs a local HTTP server that emulates the Lambda runtime. Auto-recompiles on file changes.

```bash
cd lambda
cargo lambda watch
```

In another terminal:
```bash
curl -X POST http://localhost:9000/ \
  -H "Content-Type: application/json" \
  -d '{
    "paper_text": "This study examines the effect of dropout regularization on neural network training. We found that a dropout rate of 0.5 applied to fully connected layers reduced overfitting by 15% on CIFAR-10.",
    "question": "What dropout rate did they use and what was the result?",
    "model": "sonnet"
  }'
```

### 2. sam local invoke (matches real Lambda environment)

Runs the Lambda in a Docker container that closely matches the AWS Lambda runtime. Uses the packaged SAM build artifact.

```bash
sam build
sam local invoke DocChatterFunction -e events/test-chat.json
```

### 3. curl the deployed Function URL (real deployment)

Tests the actual deployed Lambda in AWS, using the Lambda's IAM role.

```bash
curl -X POST $FUNCTION_URL \
  -H "Content-Type: application/json" \
  -d '{
    "paper_text": "This study examines the effect of dropout...",
    "question": "What dropout rate did they use?",
    "model": "sonnet"
  }'
```

### Comparison

| | cargo lambda watch | sam local invoke | curl $FUNCTION_URL |
|---|---|---|---|
| Code runs on | Your machine (native) | Your machine (Docker) | AWS |
| Credentials | ~/.aws/credentials | ~/.aws/credentials | Lambda IAM role |
| Needs deploy | No | No | Yes |
| Needs Docker | No | Yes | No |
| Auto-recompile | Yes | No | No |
| Feedback loop | Seconds | ~10s (build + container) | 30-60s (deploy first) |

### Multi-turn test

```bash
curl -X POST http://localhost:9000/ \
  -H "Content-Type: application/json" \
```

## Deploying

**First time:**
```bash
sam build
sam deploy --guided --resolve-s3
```

Use `doc-chatter` as the stack name, `us-east-1` as the region. This creates `samconfig.toml` for future deploys.

**Subsequent deploys:**
```bash
sam build && sam deploy
```

**Active development against deployed stack:**
```bash
sam sync --stack-name doc-chatter --watch
```

This watches your files and pushes code changes directly to the deployed Lambda in seconds, without a full CloudFormation update. Infrastructure changes (template.yaml) still trigger a full stack update automatically. Much faster than `sam build && sam deploy` when iterating.

### After deploying

The deploy output shows three values:

- **FunctionUrl** — the Lambda Function URL for streaming chat (`/chat`)
- **ApiUrl** — the API Gateway URL for session management (`/sessions`)
- **ApiKeyId** — the API key ID (not the value)

To get the API key value:

```bash
aws apigateway get-api-key --api-key <API_KEY_ID> --include-value --region us-east-1 --query 'value' --output text
```

Store these in `lambda/.env` for local reference (this file is gitignored):

```
API_GW_URL=https://<api-id>.execute-api.us-east-1.amazonaws.com/prod
FUNCTION_URL=https://<function-url>.lambda-url.us-east-1.on.aws/
API_GW_KEY=<api-key-value>
```

### Testing the deployed API

```bash
# Create a session
curl -s -X POST $API_GW_URL/sessions \
  -H "Content-Type: application/json" \
  -H "x-api-key: $API_GW_KEY" \
  -d '{"paper_text":"This study examines the effect of dropout...","title":"Dropout Study"}'

# List sessions
curl -s $API_GW_URL/sessions -H "x-api-key: $API_GW_KEY"

# Get a session
curl -s $API_GW_URL/sessions/$SESSION_ID -H "x-api-key: $API_GW_KEY"

# Delete a session
curl -s -X DELETE $API_GW_URL/sessions/$SESSION_ID -H "x-api-key: $API_GW_KEY"

# Chat (use session_id and token from create response)
curl -s -X POST $FUNCTION_URL/chat \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"session_id":"$SESSION_ID","question":"What did they find?"}'
```

## Reference

The Rust Lambda patterns are based on `KumoTrojanGenAIOrchestrationRust` at `/Volumes/workplace/ai-orchestration/src/KumoTrojanGenAIOrchestrationRust`. Key patterns borrowed:

- Lambda binary named `bootstrap` (AWS custom runtime convention)
- Release profile: `opt-level = "z"`, `lto = true`, `codegen-units = 1`, `strip = true`
- `aws-config` for automatic credential resolution (picks up `ada` credentials)
- `tokio` async runtime, `serde`/`serde_json` for serialization, `tracing` for logging
